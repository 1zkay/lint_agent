# 基于 LangChain、LangGraph、MCP 与 Agentic RAG 的 Verilog 智能分析系统设计与实现

## 摘要

随着 HDL 设计规模持续扩大，Verilog/SystemVerilog 工程中的代码审查工作面临多源信息分散、分析链路割裂以及标准语义难以快速核证等问题。传统 lint 工具能够高效给出违规列表，但在误报甄别、漏报补充、结构级证据关联和标准条文支撑方面仍依赖大量人工判断；与此同时，AST、控制流与数据流分析结果虽然具有较强的解释价值，却往往缺乏与自然语言交互系统的一体化整合机制。针对上述问题，本文围绕硬件代码智能审查场景，设计并实现了一套基于 LangChain、LangGraph、MCP 与 Agentic RAG 的 Verilog 智能分析系统，旨在通过“主控智能体 + 确定性分析工作流 + 标准知识检索 + 工程化持久化”的组合架构，提高复杂硬件代码问题定位、规则解释与标准核证的效率和一致性。

系统构架采用三层架构设计：1. 表示层：提供对话式交互、文件上传、任务面板、审批操作、历史会话恢复和附件管理能力，支持异步消息处理与持续会话运行；2. 业务逻辑层：以 `create_agent` 构建主控智能体，通过 `TodoListMiddleware`、`ReflectionMiddleware`、`SummarizationMiddleware`、`SkillsMiddleware`、`ToolRetryMiddleware`、`ShellToolMiddleware` 与 `HumanInTheLoopMiddleware` 等机制实现复杂任务统一编排，并以 `astream` 方式输出执行过程；3. 数据访问与知识服务层：通过 `MultiServerMCPClient`、`load_mcp_tools`、`FastMCP`、`StateGraph`、`ChainlitDataLayer` 与 `AsyncPostgresSaver` 等开放接口集成静态分析、结构解析、标准检索、关系型持久化和对象存储能力，负责完成分析任务执行、结构化证据生成、标准检索问答与结果归档管理。

本文的技术创新点主要体现在以下方面：1. 智能体与确定性工作流协同。系统将开放式交互与工程化分析分层处理，主控智能体负责工具选择与上下文协调，确定性工作流以 `StateGraph` 为基础组织分析阶段，从而兼顾灵活性与可重复性；2. 标准知识检索增强。围绕 IEEE标准文档构建 Agentic RAG 检索链路，通过 LangChain Tool 接口向上提供统一能力入口，并结合问题改写、相关性评分与页码引用机制，提升 Verilog/SystemVerilog 语义判断的可追溯性；3. 工程运行机制完善。系统引入任务列表同步、工具失败自动重试、长对话摘要压缩、高风险操作审批、历史会话持久化与附件存储等机制，使智能分析能力能够在真实项目环境中稳定运行。

在数据处理和业务逻辑优化方面，系统重点开展了以下工作：1. 建立统一输入规范，将工程工作区、项目上下文、源文件集合、lint 报告与结构分析结果纳入一致的数据准备流程；2. 构建多源证据融合机制，将源代码、违规信息、AST、控制流和数据流结果以及标准知识检索结果联合用于问题分析；3. 规范分析结果输出方式，支持面向任务场景的结构化 JSON 输出与结果校验，并通过 `BaseModel` 与结构化输出接口强化结果约束；4. 采用异步子进程执行、持久工具会话、标准化模型接入和对象存储管理等工程策略，提升系统在长时运行、复杂任务和多轮会话场景下的可用性。未来工作将围绕多角色协同智能体、离线知识库、自动化评测、安全增强与本地模型适配等方向继续展开。

**关键词：** Verilog 智能分析；硬件代码审查；LangGraph；MCP；Agentic RAG；标准知识检索

## 1. 项目背景与建设目标

硬件设计审查与软件代码审查存在显著差异。Verilog/SystemVerilog 代码不仅涉及语法与风格问题，还包含过程赋值语义、时序行为、组合逻辑完整性、接口约束、语言标准一致性等复杂因素。仅依赖 lint 报告往往难以形成高质量结论，主要原因包括三个方面：一是工具输出天然以“规则命中”为中心，缺乏对问题上下文与根因的综合解释；二是结构化分析结果与代码上下文通常分散在不同工具环节，人工整合成本高；三是对标准语义问题的判断缺乏统一的条文核证路径，容易出现结论不一致或依据不足的情况。

基于上述背景，本系统的建设目标可概括为四点：第一，在保留专业静态分析工具链能力的前提下，为硬件代码审查提供统一的自然语言交互入口；第二，将 lint 结果、源码、结构化分析结果和标准知识检索能力纳入同一分析闭环；第三，为复杂问题提供可引用、可验证、可复盘的分析依据；第四，形成能够长期运行、支持持久化、具备人机协同控制能力的工程化智能分析系统。

## 2. 系统总体架构

### 2.1 表示层

表示层负责承接用户与系统之间的所有交互过程。该层支持自然语言问答、文件上传、图文消息展示、任务进度反馈、审批交互以及历史会话恢复。对文本类附件，系统会将其转化为可直接参与推理的内容块；对图片与二进制文件，则通过统一索引方式保留可访问路径，并在需要时交由工具链继续处理。为了满足持续使用需求，表示层同时支持会话级状态管理、附件持久化与历史消息恢复，使系统从一次性工具调用扩展为可持续交互的应用形态。

### 2.2 业务逻辑层

业务逻辑层以主控智能体为核心，负责理解用户意图、组织工具调用顺序、协调不同能力单元并生成最终回答。在实现上，该层通过 `create_agent` 构建统一入口，并利用 `TodoListMiddleware`、`FilesystemMiddleware`、`ReflectionMiddleware`、`SummarizationMiddleware`、`SkillsMiddleware`、`ToolRetryMiddleware`、`ShellToolMiddleware` 与 `HumanInTheLoopMiddleware` 对执行过程进行增强，同时配合联网检索工具和请求抓取工具扩展外部能力。由此，系统能够在复杂审查任务中同时兼顾执行效率、分析质量、可解释性与安全性。

该层的一个重要特征是将开放式问题处理与确定性工程流程分离。用户请求首先由主控智能体进行任务识别与工具决策，随后根据具体任务选择进入标准知识检索路径、静态分析路径、文件处理路径或技能执行路径。这种模式既保留了大模型在任务理解和推理组织上的优势，又避免了将工程步骤完全交给大模型自由发挥所带来的不稳定性。

### 2.3 数据访问与知识服务层

数据访问与知识服务层承担底层分析执行和知识支撑职责，主要包括四类能力：1. 静态分析能力，用于批量执行硬件代码分析并生成违规报告；2. 结构理解能力，用于提取抽象语法树、控制流图、数据依赖关系与其他结构化证据；3. 标准知识能力，用于围绕 IEEE标准 提供带证据的检索式问答；4. 持久化能力，用于管理历史会话、分析状态和用户上传附件。该层通过 `FastMCP` 对外提供协议化能力，并利用 `@mcp.tool`、`@mcp.resource` 和 `@mcp.prompt` 等开放接口组织分析能力暴露形式；在持久化方面，则通过 `ChainlitDataLayer`、`AsyncPostgresSaver` 与 S3 兼容存储接口完成历史会话和附件管理。

该层通过协议化接口向上暴露资源、Prompt 和工具，使上层智能体无需关心具体工具链实现细节，即可在统一上下文中完成调用。同时，工具装配过程通过 `MultiServerMCPClient` 与 `load_mcp_tools` 完成，状态持久化则结合 `aupdate_state` 等能力维持多轮会话一致性。这种设计有效降低了多工具协同的耦合度，为后续继续扩展更多分析能力或知识源奠定了基础。

## 3. 核心能力设计

### 3.1 主控智能体编排机制

系统将主控智能体定位为“任务协调者”而非“纯回答生成器”。在每轮交互中，主控智能体通过 `create_agent` 统一接收用户输入，并结合 `astream` 持续输出消息流与状态流。主控智能体需要根据用户问题决定是否调用静态分析工具、结构分析工具、标准知识检索工具、文件工具或技能模块，并根据工具返回的结果继续组织下一步推理。为了提升复杂任务处理能力，系统引入 `TodoListMiddleware` 对多步骤任务进行显式状态管理，并借助 `ReflectionMiddleware` 对草稿回答进行质量检查，在必要时注入反馈并重新生成结果。

### 3.2 确定性分析工作流

与开放式问答不同，硬件代码基础分析更适合采用确定性工作流。系统将该类任务划分为顺序明确的多个阶段，包括静态分析执行、结构分析生成、源代码整合和产物归档，并通过 `StateGraph` 将这些阶段组织为统一流程。每个阶段都面向固定输入与固定输出，并在失败时通过条件路由提前终止。这一设计的意义在于：一方面确保分析产物来源清晰、顺序可控，另一方面也为后续智能体推理提供了稳定的数据基础。

### 3.3 结构化证据提取机制

系统不仅关注规则命中结果，还强调结构化证据对问题解释的支撑作用。通过对抽象语法结构、控制路径和数据依赖关系的提取，系统可以在后续分析中回答“问题发生在什么代码位置”“由哪条路径触发”“数据如何传播”“是否与过程语义有关”等更具解释性的审查问题。这种结构化证据机制使系统从简单的违规汇总升级为具备一定根因分析能力的智能审查工具。

### 3.4 标准知识检索与条文核证

针对 Verilog/SystemVerilog 语义判断中高度依赖标准条文的问题，系统构建了面向 IEEE标准 的 Agentic RAG 模块。该模块使用 `PyPDFLoader` 进行标准文档分页读取，利用 `RecursiveCharacterTextSplitter` 完成文本分块，使用 `OpenAIEmbeddings` 与 `FAISS` 构建向量索引，并在运行时通过 `bind_tools`、`ToolNode`、`tools_condition` 和 `StateGraph` 组织“检索 - 评分 - 改写 - 回答”的链式策略提升检索效果；在相关性判定阶段，则结合 `BaseModel` 与 `with_structured_output` 实现结构化输出约束。对外则通过 LangChain Tool 接口统一暴露标准检索能力。对于涉及变量类型、赋值语义、调度规则、过程块约束、接口或断言等问题的查询，系统能够返回中文回答并附带页码级引用，从而显著提高问题结论的可验证性。

### 3.5 技能扩展与标准化输出

在面向特定场景的复杂任务中，系统进一步引入技能机制，将任务流程、输入要求、输出约束和结果校验逻辑封装为可按需加载的领域能力单元。以 lint triage 场景为例，系统能够围绕源文件、lint 报告、规则知识和标准知识执行组合分析，并输出结构化 JSON 结果。技能机制的优势在于其可维护性和可扩展性较强，适合在不大幅修改主系统逻辑的前提下，逐步接入更多面向硬件设计场景的专项分析能力。

## 4. 技术创新与工程优化

### 4.1 智能体与工具链协同创新

相较于传统“脚本串联工具”的实现方式，本系统的一个重要创新点在于引入智能体作为调度中枢，同时保留分析链路中的确定性步骤。主控智能体以 `create_agent` 为统一调度入口，工作流以 `StateGraph` 为执行框架，两者结合后既保留了自然语言交互的灵活性，也保证了底层分析链路的稳定性和可复现性。

### 4.2 多源知识融合创新

系统并未将“工程规则知识”和“语言标准知识”混为一体，而是分别处理、协同使用。前者更偏向设计规范、经验规则和问题归类，后者更偏向语言语义和标准条文核证。通过这种双知识源融合机制，系统能够在工程实践与标准依据之间建立更清晰的映射关系，避免仅凭经验规则或仅凭标准条文作出片面判断。

### 4.3 工程运行机制优化

为了支撑真实项目场景，系统在运行层面引入了多项优化策略，包括基于 `astream` 的异步消息处理、基于 `aupdate_state` 的状态一致性维护、对象存储附件管理、高风险工具审批、工具失败重试以及长上下文自动压缩等。这些设计使系统不再停留在实验性原型层面，而是具备了可持续运行、可恢复、可审计的应用特征。

### 4.4 分析结果规范化优化

针对复杂审查任务往往输出松散、不易复用的问题，系统强调分析结果的规范化表达。一方面，通过统一的数据准备和归档流程，确保不同来源的分析产物具有稳定结构；另一方面，通过技能化输出与结果校验机制，推动重点任务结果以标准化 JSON 形式沉淀，方便后续自动处理、比对和复查。

## 5. 数据处理与业务逻辑优化

参考专业领域问答系统的设计思路，本系统在业务逻辑上重点强化了五项能力：1. 统一输入建模，将用户请求、工程上下文、分析产物和标准检索结果纳入同一任务语义空间；2. 规范结果展示方式，使问答输出、任务进度、结构化结论和证据引用能够相互对应；3. 强化时间与状态管理，保证多轮对话、任务执行和历史恢复之间的一致性；4. 建立可扩展的知识装载机制，使规则知识、标准知识和场景技能能够按需接入；5. 优化异常处理链路，在网络波动、工具失败或依赖异常时通过重试、回退和审批机制降低失败对整体体验的影响。

## 6. 未来工作

未来可从以下几个方面继续完善。第一，进一步由“主控智能体 + 专项能力单元”演进到更加明确的多角色协同架构，例如拆分为规则判责、标准核证、修复建议与结果审校等角色；第二，扩展知识体系，将语言标准、设计规范、项目历史缺陷和企业内部经验规则纳入统一知识框架；第三，补齐自动化评测体系，为问题识别准确率、条文引用准确率和修复建议质量建立离线评估机制；第四，增强离线部署能力和本地模型适配能力，以满足对安全性、内网运行和资源可控性要求更高的应用环境。

总体而言，该系统已经具备从“静态分析工具封装”演进为“硬件代码智能分析平台”的基础条件。其在自然语言交互、确定性工作流、标准知识检索与工程化运行之间形成的协同关系，为专业领域智能分析系统的设计与实现提供了可借鉴的实践路径。
